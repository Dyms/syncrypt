# Localization Policy

## Canonical language

**English is the single normative source of truth** for all documentation, and is
the language of code, identifiers, and the RFC/ADR record. This keeps one
authoritative version and avoids drift — consistent with the project ethos
("no hidden state", "never surprise").

## What we translate, and when

| Layer | Localized? | When |
|---|---|---|
| **User-facing** (README, `user-guide/*`) | Yes → `docs/ru/` | Now; kept reasonably current |
| **RFCs / ADRs** (normative spec) | Not yet | After the spec stabilizes (tag `spec-v1.0`) — translating a moving target guarantees drift |
| **Code / `.ai/` agent context** | English only | Always (implementation language) |

Rationale (architect's note): the architecture is still actively evolving. Fully
translating the normative spec now would double every future edit and risk the
translation silently diverging from the source — the exact kind of hidden
inconsistency the project is built to avoid. We localize the stable, user-facing
surface now and translate the spec once it is frozen.

## Rules for translations

- Every translated file carries a banner: *"Перевод. Источник истины — английская
  версия; при расхождении верна английская."* with a link to the original.
- Translations may lag. If a RU page looks stale, the EN page wins.
- Do not fork meaning: a translation restates, it does not add or change decisions.

## Adding a language

Mirror the `user-guide` tree under `docs/<lang>/` (e.g. `docs/ru/`). Keep filenames
identical to the English originals so cross-links map 1:1.
